# 星形线案例设计沉淀

核心场景：一根棍子靠墙缓慢滑落，扫过区域的边界是什么。

最终叙事目标：让观众先看见“很多瞬间叠在一起形成外轮廓”，再自然接受这条边界是包络线，并通过微分推导得到星形线方程。

## 内容路径

1. 用墙角棍子滑落建立生活场景，不先讲公式。
2. 给画面里的变量起名字：棍长 `L`、夹角 `θ`、地面截距 `a=Lcosθ`、墙上截距 `b=Lsinθ`、棍子上的点 `P=(x,y)`。
3. 从截距式直线 `x/a + y/b = 1` 得到 `x secθ + y cscθ = L`，说明 `sec` 和 `csc` 只是倒数记号。
4. 用“相邻几根棍子一起擦出边界点”解释包络线条件：`F=0` 和 `∂F/∂θ=0`。
5. 明确导数来源：`d(secθ)/dθ = secθ tanθ`，`d(cscθ)/dθ = -cscθ cotθ`，因此得到 `x secθ tanθ - y cscθ cotθ = 0`。
6. 展示联立过程，而不是跳步：从导数条件化成 `x sin^3θ = y cos^3θ`，再得到 `x=Lcos^3θ, y=Lsin^3θ`。
7. 最后用自然排版展示 `x^(2/3)+y^(2/3)=L^(2/3)`。
8. 收尾画四象限完整星形线，让观众直观看到它为什么叫“星形线”，再补充机械臂边界、焦散、安全包络等现实影子。

## 已采纳的改进

- 开头像讲故事一样引出背景，不一开始提公式、定理或“第几章”。
- 旁白像对着一个人轻松讲事，少用生硬标题和书面化提示。
- 画面标题只做章节概述，不出现“先把字母绑回画面”这类不自然表达。
- 变量必须和真实物理量绑定，尤其是 `θ`、`a=Lcosθ`、`b=Lsinθ`、`P=(x,y)`。
- 推导区同时出现多条表达式时，除非有精确到表达式的旁白 cue map，否则不要强行高亮公式；当前版本优先通过出现顺序、留白和颜色降低理解成本。
- `θ` 标识放在地面端点 A，表示地面方向到棍子的夹角。
- `x secθ + y cscθ = L` 出现前解释 `sec/csc` 的来源。
- `x secθ tanθ - y cscθ cotθ = 0` 必须说明来自对 `F` 关于 `θ` 求导，且求导时固定 `x,y`。
- 从导数条件到参数式不能突兀，要展示中间比例关系。
- 画面和配音必须按 SRT 时间轴对齐。
- 幂次尽量使用自然数学排版，例如竖式 `\frac{2}{3}`。
- 中文使用清晰字体，西文、数字和公式变量尽量使用 Times 系列。
- 视频需要短封面帧，网页预览需要显式封面图，避免黑屏缩略图。
- 本地网页需要标题、简介、标签、状态、封面和章节进度；章节进度靠近播放器，并支持点击跳转。
- 低清预览确认后再渲染最终视频。

## 踩坑记录

- 自动抽帧封面可能抽到黑屏或淡入前空画面，重要视频应在 `data/videos.json` 配置 `cover`。
- Edge TTS 可能把 `sin/cos/sec/csc/tan/cot` 读错，旁白稿可写 `sine/cosine/secant/cosecant/tangent/cotangent`，画面公式仍显示标准缩写。
- PowerShell 控制台可能把 UTF-8 中文显示成乱码；检查源码和 JSON 时优先用 Python 以 `encoding='utf-8'` 读取。
- 当前项目本地 FFmpeg 只有 `ffmpeg.exe`，不一定有 `ffprobe.exe`；查时长可解析 `ffmpeg -i` 输出。
- MiKTeX 初装可能缺 Times 风格数学包；本项目用 `mathptmx` 作为 Times 系数学模板。
- Manim CLI 没有直接提供“一个命令内多线程渲染多个 Scene”的参数；可用多进程并行渲染不同 Scene。
- 并行渲染不要开太高；LaTeX、SVG 缓存和磁盘写入会抢资源，低清预览默认 `MaxParallel 3` 较稳。
- 章节越长越难定位节奏问题，复杂推导拆成小场景更利于局部重渲和时间校准。
- 页面布局应服务检查流程：播放器优先、章节进度紧贴视频、元数据靠后、片库用于快速索引。
