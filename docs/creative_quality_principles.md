# 创作质量原则

这份文档沉淀用户在视频迭代中提出的可复用改进建议，用于提升后续中文数学科普动画质量。

## 开头与叙事

- 从具体场景、反常识现象或自然设问开始，不要一上来讲公式、定理或“第几章”。
- 开头不要假设观众已经在想某个公式；让疑问从画面和故事里自然长出来。
- 如果开头说某个角色“只按简单规则行动”，必须立刻说清可观察的具体规则；不能先让画面运行一段时间，只把“简单规则”当作抽象标签。
- 公式类视频可以从“只背现成答案会在哪里卡住”切入，再提出可计算、可验证的问题；不要使用“公式长出来”这类拟人化但不解释机制的说法。
- 章节标题用于概述内容，不暴露制作过程，不使用“先把字母绑回画面”这类不自然表达。
- 同一个术语如果已经在前文出现过，后文要承接它已经被提到的事实，说明它在当前推导中承担的角色，不要再次像第一次介绍。
- 新专有名词第一次出现时，旁白要立刻给出一句可操作的基础定义；后续专门章节应扩展它的背景、用途或相关领域，而不是重复首次介绍或强行和前文建立单一关系。
- 故事桥段不能把未来解释提前塞给观众：不要在对象出场前说它的名字、目的、结论或之后才会出现的抽象含义；也不要用“先不看 X”否定一个尚未出现的概念。
- 比较句必须有同一条比较轴。不能把“越走越乱”和“把灯收齐”这类不同性质的描述并列；应改成“同样是短规则，一个让纹理铺开，另一个让变化可追踪”这类同轴表达。
- 当核心承诺发生变化时，要果断删掉不再服务主题的素材。漂亮的角色、机制或支线如果把视频从“混沌”带向“秩序/计算”，应保留到另一个话题，而不是硬塞进当前成片。
- 收尾要展示完整视觉 payoff，例如完整曲线、现实应用或记忆锚点，而不只是停在方程。

## 变量与推导

- 新变量出现前，必须先和真实物理量或画面对象建立联系。
- 组合变量或新记号不能突然出现；要解释它们来自哪里、代表什么。
- 推导中的关键结论要说明来源，特别是求导条件、固定量、联立步骤和参数式转化。
- 推导开始前要先说清当前正在解决的具体问题，例如“求这条反射光线的直线方程”或“找这一族线的包络条件”。不能只用“来源”“条件”“下一步”这类抽象标题让观众猜。
- 包络条件里的 `\partial F/\partial t=0` 不能说成“对 `F=0` 求导”。画面可以保留简洁的 `F=0` 和 `\partial F/\partial t=0` 两条件，但旁白或小字必须说明：先固定候选点 `Q=(x,y)`，只让族参数 `t` 变化；这个偏导描述的是邻近光线是否也贴着 Q 擦过。
- 向量反射公式不能直接跳到压缩结果。像 `\vec{d}=\vec{i}-2(\vec{i}\cdot\vec{n})\vec{n}` 这种式子，应先展示“法线投影”和“切线分量”；如果已知向量能让投影系数马上化简，例如 `\vec{i}\cdot\vec{n}=\cos t`，就在投影步骤先化简，再说明切线分量保持不变、法线分量反向，最后合并成紧凑公式。
- 当画面出现一串特殊系数或模式时，不能只标注名称；要给出至少一个可见来源或计数理由，避免观众把规律理解成巧合。
- 不要用尚未推导出的数字、系数或结论做段落入口；先给观众看到它从哪一步算出来，再命名和抽象。
- 当公式从逐项等式切换到求和记号或压缩记号时，必须展示中间操作和定义替换，例如“等式两边同时求和”再“按定义改写成 S_p(n)”。
- 当向量式、坐标式和三角恒等式连续转换时，不能从抽象向量式直接跳到最终坐标。先代入坐标并展开，例如从 `\vec{i}-2\cos t\,\vec{n}` 到 `(1-2\cos^2 t,-2\sin t\cos t)`，再使用二倍角公式。
- 隐式直线方程不能突然出现。如果一条式子来自“过某点、沿某方向”的直线，要先展示点、方向、垂直辅助向量或等价的行列式关系，再展开成最终方程；辅助向量如果出现在公式里，图上也要标出来。
- 新公式形式要说明用途，不只是说明来源。比如把单条光线写成 `F(x,y,t)=0` 时，要告诉观众这是为了把所有 `t` 的光线统一成一族方程，下一步才能找包络，而不是为了多写一个抽象符号。
- 从包络条件解出参数曲线时，不能从 `F=0` 和 `\partial F/\partial t=0` 直接跳到最终 `x(t),y(t)`。要先把原方程和偏导方程写成关于 `x,y` 的方程组，说明 `R,t` 暂时作为参数，再展示至少一个中间坐标式，最后才化简成参数式。
- 命名曲线解出来以后，如果补充参数式、直角坐标式或极坐标式，必须先交代坐标约定和变量含义。极坐标尤其要在图上标出极点、极轴、径向距离和方向角，并说明这只是同一条曲线的另一种写法。
- 漂亮但偏离主线的结论只作为彩蛋或一句提及；如果它迫使观众理解另一套图示关系，就会稀释主推导，应压缩或删除。
- 抽象步骤要配一个形象比喻或物理锚点，让观众知道“为什么要这么做”。
- 没有精确到表达式的旁白时间 cue 时，不要强行给公式做高亮；错位高亮比不高亮更干扰理解，优先靠出现顺序、留白和颜色层级解决指向问题。

## 画面与排版

- 每一屏只突出一个认知焦点，避免文字、公式和图形互相抢位置。
- 后续数学科普视频默认沿用贝塞尔成片确认过的深色背景配浅网格视觉系统：网格线 `#8FA8D8`、`stroke_width=1.6`、`opacity=0.155`、`spacing=0.48`。如主题需要改色，也先生成单帧候选并经确认后再重渲染整片。
- 当一张核心图承载超过几秒旁白时，画面必须保留与语义相关的轻微运动，例如控制点拖动、参数扫过、权重变化或局部高亮；不要让观众在旁白解释期间长时间看静态图。
- 字幕和公式主体要错位摆放，防止重叠。
- 图形比喻必须能从画面直接读出含义；不要为了“有图标”而放生硬的机器、齿轮或抽象符号。
- 专业术语型比喻如果不直观，优先换成过程画面本身，例如逐项抵消、逐步替换、首尾保留，而不是要求观众先理解比喻。
- 用块、条、括号、箭头表示数学对象时，必须标清每个图形代表的是“单项”“前几项累计”还是“总和”。
- 动画转化要保留可见关系；如果从一个对象变成另一个对象，画面要用箭头、标签或中间状态说明两者为什么相关。
- 旁白声称“只差一个初始条件”“多一个标记”“某处发生扰动”时，画面必须把这个差异显性标出；不要指望观众从密集网格或复杂纹理里自己找出一个像素级差别。
- 角色或吉祥物只服务理解，不应挡住主要对象。调整大小、位置和图层时要优先保证棋盘、灯泡、公式或关键标记可读。
- 如果行为发生在网格上，角色位置必须严格落在格子中心，留下的痕迹必须对齐格线；否则规则迭代会看起来像普通随机动画。
- 快速迭代场景要预设足够大的视野，只有接近边界时才平滑拉远；不能让镜头频繁追随抖动，也不能在拉远后改变网格尺度导致“格子变大/对不上”。
- 对齐和边界是低清预览的硬指标：柱状图底线、说明文字左边界、公式栈底部都要抽帧检查，不能靠肉眼猜。
- 数学幂次和分数尽量使用自然排版，例如竖式 `\frac{2}{3}`，不要用横写 `2/3` 糊弄。
- 中文默认使用项目本地的得意黑 / Smiley Sans 字体资产 `assets/fonts/SmileySans-Oblique.ttf`，延续贝塞尔视频确认过的字体系统；除非字体文件缺失或主题有明确风格理由，不要退回 `Microsoft YaHei`。完整公式优先使用 `MathTex`。中文说明句里不要盲目用 `t2f` 把行内字母数字切到 Times 系列，容易出现基线偏移或 Pango 样式冲突；需要专业数学字形时，拆成 `Text` 与 `MathTex` 手动对齐。
- 右侧信息面板以后以咖啡杯焦散线这版为最小字号标准：`PANEL_TITLE_SIZE >= 0.42`，`PANEL_BODY_SIZE >= 0.34`，`PANEL_NOTE_SIZE >= 0.34`，`PANEL_FORMULA_SIZE >= 0.66`，`PANEL_FORMULA_LARGE_SIZE >= 0.72`，`PANEL_FORMULA_SMALL_SIZE >= 0.54`。移动端可读性优先于桌面端“看起来精致”；如果装不下，先拆分内容或分步展示，不要直接把字号压回去。
- 公式栈如果连续几行左侧表达式相同，后续行可以只保留左对齐的等号和右侧结果，减少重复阅读负担；等号位置要稳定对齐。
- 角度、截距、点和曲线标签的位置必须符合实际几何含义，不能只是“看起来差不多”。
- 公式或面板里出现的几何符号、向量符号必须和画面中的对象同步标识；例如右侧写 `\vec{i}`、`\vec{n}`、`\vec{d}` 时，左侧对应的入射方向、法线方向和反射方向也要用同色标签标出来。
- 全片预览前要做坐标参照巡检：凡是画面出现 `x,y` 坐标公式，左侧图必须有直角坐标轴或等价基底；凡是出现极坐标方程，必须有极点、极轴、径向线段和角度提示。结果展示段也不能例外。
- 展示反射光线族时，至少保留一条动态高亮的“当前入射光线”和对应的“当前反射光线”。只显示一组淡入射背景和一条高亮反射线，会让观众看不出反射关系。
- 凡是教学画面涉及入射光线或反射光线，优先在高亮光线或代表光线上加方向箭头；密集光线束可以只给当前光线或一条代表光线加短箭头，重点是让观众看出光从哪里来、往哪里走。开场或结果展示如果主旨是“叠加出的亮边界”，不要为了补箭头而给某一条反射线加抢眼标记，以免观众误以为那条线是重点。
- 侧栏图例或说明列表描述几何对象时，也要同时给出符号和含义，例如 `O：圆心`、`R：半径`、`t：角度`、`OP：法线`；只写“圆心/半径/角度”会让观众重新在图和公式之间猜对应关系。
- 向量坐标或坐标公式出现时，图形区必须同时给出 `x/y` 坐标轴或基底参照；公式面板中的公式栈要作为整体在容器里居中或按明确网格对齐，不能让容器一侧留下无意义的大空白。
- 参数驱动的 UI、代码面板或仪表盘如果用非 Manim 渲染，不能让它接管整屏视觉系统；Manim 仍应负责全局背景、标题、底部公式和数学字体，外部渲染器只负责局部动态内容。

## 旁白与声音

- 旁白像对一个好奇的人轻松讲故事，少用书面化、命令式或制作流程式话语。
- 成片开头要留出短暂呼吸，通常让第一句旁白在画面出现约 0.8 到 1.2 秒后再进入。
- 如果开头已经有独立品牌标识或制作说明，旁白进入点必须等标识完成后再进入正题，不能让下一段声音压在上一段画面上。
- 配音不需要念“第几章”，章节感交给画面和网页进度条。
- TTS 容易读错 `sin/cos/sec/csc/tan/cot` 时，旁白稿写成 `sine/cosine/secant/cosecant/tangent/cotangent`，画面公式仍显示标准缩写。
- TTS 遇到“量”等多音字时，不要孤立使用容易误判的字；优先改写成“变量”“物理量”“数值”“长度”等更明确说法。
- 画面进度必须跟配音长度对齐，不能出现声音进入下一段而画面还停在上一段。
- 旁白说“换一间房间”“现在看右边”“跑到一百万步”等具体画面状态时，当前帧必须已经呈现对应对象或数值；这类句子需要抽关键帧验证，而不只是看总时长。

## 预览与封面

- 每个重要视频需要短封面帧，保证网页和播放器预览不是黑屏。
- 封面帧只承担预览功能，不应占用过长时间；标题要大、可读、有吸引力。
- 封面标题必须按手机双列视频流缩略图的可读性设计：约 160 到 200 像素宽的预览里，主标题大字仍应基本可读；但默认只落盘保存一张高清封面图，不额外生成 200px 检查图。
- 封面主标题优先使用短句，最多两行；不要把真正的标题压成小字，也不要依赖副标题解释主题。
- 主标题应占据首屏封面的主要视觉权重，使用高对比颜色、粗壮字重和足够行距；复杂公式、细线图形和长副标题不能抢标题可读性。
- 制作封面时不要只靠主观 `scale()` 倍率；更稳妥的是按目标宽度铺标题，让标题实际占据画面宽度的一半以上。
- 封面构图要主动吃满画面：主标题和一个核心视觉应形成主要面积，不能用小面板、小副标题或低透明水印留下大块无效空白。
- 封面安全边距要足够，避免标题贴边、被播放器时长角标遮挡，或在平台裁切后丢失关键词。
- 封面不再依赖正片第一帧；默认单独渲染或生成一张封面图，以手机端/双列信息流的可读性为准。
- 除非用户明确要求多平台差异版，不再默认分别制作电脑端和移动端封面；宽屏预览也复用这张移动端优先封面。
- 封面 JPG 存放在 `topics/<topic>/exports/covers`，由 `data/videos.json` 的 `cover` 引用；如保留 `covers.desktop` 与 `covers.mobile`，默认让二者指向同一张封面图，不作为源码提交。
- 低清预览通过后再渲染高清，先验证叙事、推导、配音、封面和章节跳转。
- 本地图库必须展示可读标题、简介、标签、状态、封面和章节进度，章节控件靠近播放器。

## 工作流

- 复杂视频拆成多个章节 Scene，独立低清渲染后再拼接。
- 只修改片头、片尾或单个章节时，优先用图库 `segments` 分段预览把多个低清片段当作完整视频检查，避免每次都先生成完整 MP4。
- 用 SRT 时间轴校准每个 Scene 的停顿。
- 全局背景、网格、卡片透明度这类视觉系统参数，先生成单帧预览并使用唯一文件名确认；不要在没有静帧确认的情况下直接重渲染整条视频。
- 并行渲染可以加速，但不要过高；LaTeX、SVG 缓存和磁盘写入会抢资源。含动态 `Text`/`MathTex` 的场景如果复用同一个 Manim `media_dir`，优先串行渲染或隔离每个场景的缓存目录。
- 在 Windows/PowerShell 环境里，不要通过 PowerShell 命令行、stdin 管道或 here-string 写入包含中文的源码、JSON、Markdown 或脚本文本；这些路径容易把 UTF-8 中文变成问号或乱码。小改动用 `apply_patch`，程序化改动用已有 UTF-8 文件、Unicode 转义，或先写临时 UTF-8 脚本文件再执行；写完必须用 Python 读取并检查关键中文字段。
- 迭代过程中保留源码版本，清理旧视频产物，避免磁盘被临时文件占满。
- 最终版本通过后，图库 metadata 要改成面向发布的标题、简介、标签和状态，移除“v10”“低清预览”“字体试用”等迭代语义，并清理旧预览产物。
- 最终清理后必须通过本地服务接口确认同一话题只展示当前目标版本；只改 `data/videos.json` 不够，旧 MP4 留在 `exports/final` 仍会被图库自动扫描出来。
- GPT 品牌片头默认复用已批准片头，不重新设计颜色、布局、文案、时长或动画；只有用户明确要求重做时才新建片头。

## 物理现象到曲线推导

- 光学、包络线、轨迹和参数曲线类视频，要先让观众看见现象，再建立生成对象，最后才命名曲线。不要让专业名词或最终方程抢在可见机制前面出现。
- 从现象走到公式族时，必须先说明“正在求哪一个几何对象的方程”。例如反射光线题材要先交代正在求单条反射光线的直线方程，再说明把角度连续变化后会得到一族直线。
- 公式族的统一写法要说明用途：把每条线写成 `F(x,y,\theta)=0` 不是为了抽象，而是为了下一步使用包络线条件。凡是改写成隐式方程、参数族、矩阵式或极坐标式，都要说清它让下一步变得可做。
- 包络线条件必须讲清“固定点”和“变化参数”。显示 `F=0` 与 `\partial F/\partial\theta=0` 时，要说明候选点 `Q=(x,y)` 固定，族参数变化，偏导检测邻近曲线是否也贴着同一点擦过。
- 曲线命名后的 payoff 不应停在参数式。优先补充完整形状、一种机械或几何生成方式，以及必要的坐标方程；补充方程时先说明坐标约定和它与前一结果是同一条曲线的不同写法。
- 光学场景的方向箭头只服务当前语义。解释入射和反射时要有运动方向提示；进入点法式、包络条件或其它数学任务后，旧的方向箭头和标签要淡出，避免和新向量符号冲突。
- 最终清理不只清旧视频，也要清旧封面。每个最终话题默认只保留最终 MP4 和 metadata 引用的最终封面，旧预览封面、封面候选图和孤儿 poster 都应移除，除非用户明确要求保存。
